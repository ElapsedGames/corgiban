## Summary

parseLabInput() uses takeFirst() for XSB, SOK 0.17, and SLC collections. A user can paste a
multi-level document, get a successful parse, and then have normalization or export keep only the
first level. Silent truncation is the problem regardless of whether "first level only" is an
acceptable product constraint.

## Expected

Either: reject multi-level input with an explicit error, or surface an explicit "first level only"
warning in the UI so the user knows what happened.

## Actual

Multi-level input parses successfully and silently truncates to the first level.

## Repro

1. Paste a multi-level XSB collection into the /lab editor
2. Observe successful parse with no warning
3. Export or inspect -- only first level is present

## Notes

Sources: Review 5 #4 and Principal/QA review.

- `apps/web/app/lab/labFormat.ts:89-127` uses `takeFirst()` for XSB, SOK 0.17, and SLC imports.
- `apps/web/app/lab/LabEditorPanel.tsx:61-64` / `apps/web/app/lab/useLabOrchestration.ts:151-160`
  surface a normal success path with no truncation notice.
- Tests currently cover only single-level happy paths, so the ambiguity is undocumented.

## Fix Plan

Detect multi-level input and either reject with a clear error or display an explicit truncation
warning. Add a test for multi-level input to document the contract in both parsing and UI flows.
