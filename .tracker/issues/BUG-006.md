## Summary

The solved branch computes solvedFrontier but never updates maxFrontier before metrics are
emitted. Persisted benchmark records understate frontier size exactly in the success case that
Phase 6 is designed to measure and compare.

## Expected

maxFrontier should reflect the actual maximum frontier size at solve time, including the solved
branch.

## Actual

maxFrontier is not updated in the solved branch, so benchmark records understate it.

## Repro

Run a known solvable puzzle and inspect the emitted maxFrontier metric.

## Notes

Sources: Review 2 and Principal/QA review.

- `packages/solver/src/algorithms/bfsPush.ts:149-158` computes `solvedFrontier` but emits the old
  `maxFrontier`.
- `packages/solver/src/algorithms/bfsPush.ts:214-216` only updates `maxFrontier` in the later
  non-solved loop path.
- This affects surfaced benchmark metrics in the exact success cases Phase 6 comparison/export is
  meant to support.

## Fix Plan

- Update maxFrontier in the solved branch before emitting metrics
- Add a unit test asserting solved runs report the correct frontier metric
