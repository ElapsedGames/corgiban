## Summary

Documentation still calls out canvas distortion on small screens as deferred work, and the current
implementation supports that concern: `GameCanvas` defaults to a fixed `cellSize={32}` on `/play`,
with no responsive sizing policy or viewport-based fallback.

## Expected

The game board should adapt its cell size or layout constraints so smaller viewports do not distort
or overflow the primary play surface.

## Actual

`PlayPage` renders `GameCanvas` with a fixed 32px cell size, so board sizing remains static even
though project docs note small-screen distortion as an unresolved follow-up.

## Repro

1. Open `/play` on a narrow viewport or mobile-sized window
2. Load a wider level
3. Observe that the canvas uses the fixed 32px cell size rather than a responsive sizing strategy

## Notes

Documented in `docs/project-plan.md` as a deferred note. Current code references:
`apps/web/app/play/PlayPage.tsx` and `apps/web/app/canvas/GameCanvas.tsx`.

## Fix Plan

- Define a responsive cell-size policy for `/play`
- Apply it before rendering `GameCanvas`
- Add coverage for narrow viewport behavior and sizing invariants

## Resolution

- `apps/web/app/canvas/GameCanvas.tsx` now measures the available container width and clamps the
  effective cell size with `resolveResponsiveCellSize(...)`, so `/play` can shrink the board on
  narrow viewports without changing the route API.
- The canvas render path keeps CSS sizing responsive (`width`, `max-width`, `height`) while DPR
  and container width change.
- Tests: `GameCanvas.test.tsx` and `GameCanvas.effects.test.tsx` cover the responsive cell-size
  helpers and container-width subscription behavior.
