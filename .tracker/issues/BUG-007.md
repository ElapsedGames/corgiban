## Summary

spriteAtlas.client.ts creates and terminates a new worker on every atlas request. ADR-0024 and
Phase 6 intent describe a dedicated auxiliary worker. Worker startup and teardown cost is now part
of atlas generation, wiping out the intended performance win on route remounts, DPR changes, and
first paint.

## Expected

A persistent dedicated worker that is reused across atlas requests.

## Actual

New worker created and terminated per request.

## Repro

Observe worker lifecycle in devtools on route remount or DPR change.

## Notes

Sources: Review 2 and Principal/QA review.

- `apps/web/app/canvas/spriteAtlas.client.ts:140-150` creates a worker inside each request and
  terminates it after response/error.
- `apps/web/app/canvas/__tests__/spriteAtlas.client.test.ts:152-160` currently locks in that
  one-shot worker lifecycle.
- This is both a performance miss and a docs/ADR drift issue because ADR-0024 and the Phase 6 plan
  describe a dedicated auxiliary worker.

## Fix Plan

- Preferred: refactor spriteAtlas.client.ts to hold a persistent worker and reuse it across
  requests, then update tests to match
- Fallback if one-shot workers are intentional: narrow the ADR/docs language so they stop claiming
  a dedicated worker and update tests/docs consistently

## Resolution

Reused one dedicated sprite-atlas worker across atlas requests and updated canvas regression tests to cover the shared-worker lifecycle.
