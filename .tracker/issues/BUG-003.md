## Summary

The parser accepts warmup: true on imported records. importBenchmarkReport() persists whatever
parseBenchmarkReportJson() returns. But toSuiteAnalytics() groups every record without filtering
warmup === true rows. Warmup rows count toward runs, success rate, and quantiles, violating the
Phase 6 "warm-up excluded from measured results" contract.

## Expected

warmup === true rows should be excluded from analytics grouping.

## Actual

Warmup rows are counted as normal comparable suite results, which can distort success rate,
p50/p95 analytics, and exported comparison snapshots.

## Repro

1. Create a one-record report with warmup: true
2. Import it via the bench UI
3. Observe analytics render it as a normal comparable suite entry

## Notes

Sources: Review 1 #2 and Principal/QA review.

- `packages/benchmarks/src/model/benchmarkSchema.ts:246-267` accepts imported `warmup`.
- `apps/web/app/state/benchThunks.ts:272-280` persists imported records unchanged.
- `apps/web/app/bench/benchmarkAnalytics.ts:124-160` groups all suite rows without filtering
  `warmup === true`.
- `apps/web/app/bench/BenchmarkSuiteBuilder.tsx:118-125` says warm-up runs are not stored,
  so this is now a contract mismatch rather than just a display bug.
- Current analytics tests do not cover imported warm-up contamination.

## Fix Plan

- Filter warmup === true rows before grouping in toSuiteAnalytics()
- Ensure comparison helpers and snapshot export use measured rows only
- Add regression test covering warmup-row import scenario
