## Summary

In detectUnsupportedVariants(), the generic alphabetic-token branch ([a-zA-Z]) runs before the
explicit hexoban-token check. Lowercase v matches the alphabetic branch and is tagged as multiban
instead of hexoban, producing the wrong diagnostic for valid XSB/SOK/SLC inputs.

## Expected

Lowercase v should be detected as a hexoban token with a hexoban diagnostic.

## Actual

Lowercase v is tagged as multiban. Current tests cover `<` but not lowercase `v`, so the
misclassification escaped the Phase 6 variant-detection coverage.

## Repro

1. Pass an XSB grid containing lowercase v to detectUnsupportedVariants()
2. Observe wrong variant diagnostic

## Notes

Sources: Review 1 #3, Review 5 #1. normalizeGrid.ts:74-97. Phase 6 explicitly promised
unsupported-variant detection (project-plan.md:806).

## Fix Plan

- Check explicit hexoban tokens before the generic alphabetic branch
- Add regression test for lowercase `v` in normalizeGrid.test.ts

## Resolution

Reordered unsupported-variant detection so explicit hexoban tokens win before the generic alphabetic multiban branch, and added regression coverage for lowercase v plus mixed-variant deduping.
