## Summary

bfsPush.ts reads context.nowMs via non-null assertion (!). The public solve() entry point at
solve.ts:75 uses a defensive fallback for missing nowMs. Any caller that invokes solveBfsPush
directly with a partially populated context (tests, direct imports, future internal callers) will
crash with a non-null assertion error rather than failing gracefully.

## Expected

Algorithm layer should apply the same clock-unavailable contract as the solve() entry point:
fail explicitly if no monotonic clock is available, never silently use a non-monotonic source.

## Actual

context.nowMs! crashes callers that do not populate the field.

## Repro

1. Call solveBfsPush with a context object that omits nowMs
2. Observe crash at bfsPush.ts:72

## Notes

Source: Review 4 #1. Do NOT substitute Date.now() -- per docs/review-notes.md, missing monotonic
clock is an explicit failure contract.

## Fix Plan

- Remove the non-null assertion
- Apply the same defensive pattern as solve.ts:75 (use performance.now() if available, else fail)
- Add a unit test: call solveBfsPush with empty context, assert graceful failure not crash

## Resolution

Shared solver clock resolution between solve() and solveBfsPush, returning an explicit error when no monotonic clock is available and covering the direct-call path with a regression test.
