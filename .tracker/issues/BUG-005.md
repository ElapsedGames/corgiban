## Summary

renderCanvasFrame() only updates canvas.width/height and canvas.style.width/height when
plan.pixelWidth or plan.pixelHeight changes. If cellSize and dpr change in opposite directions
such that backing-store pixel size stays the same but logical CSS size changes, the canvas DOM
size stays stale.

## Expected

CSS size should always reflect the logical display size, independent of backing-store size.

## Actual

Example: 10 cells at cellSize=32, dpr=2 -> backing 640, CSS 320px. Switching to cellSize=64,
dpr=1 keeps backing 640 but CSS should become 640px. Current code leaves it at 320px.

## Repro

Simulate the cellSize/dpr change described above in a test environment.

## Notes

Source: Review 5 #2. GameCanvas.tsx:64-71. Tests at GameCanvas.effects.test.tsx do not cover
this invariant; current coverage only checks initial sizing behavior.

## Fix Plan

Update CSS size every render, or track logical size separately from backing-store size.

- Add a regression test where logical CSS size changes while backing-store pixel size remains equal
