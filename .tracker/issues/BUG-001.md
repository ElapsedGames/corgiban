## Summary

Two related failure modes in the sticky-fallback system in benchmarkStorage.client.ts. Data loss
path: syncResultsFromRepository() only assigns memoryResults after deleteRuns() resolves. If
IndexedDB reads succeed but the retention-delete step throws, storage flips to memory-fallback
with stale or empty memoryResults, losing all loaded history for the session. False-success path:
when persistence falls back to memory, save/replace/clear operations continue with a non-throwing
success-like flow, making it look like data was durably saved when it was not.

## Expected

If deleteRuns() throws after rows are loaded, the loaded rows should still be available in
memoryResults. Memory-fallback operations should not appear identical to durable-storage success.

## Actual

deleteRuns() failure causes init() to flip to memory-fallback with empty memoryResults, losing
all previously loaded history for the session.

## Repro

1. Mock repo to return 3 runs from loadResults() but throw from deleteRuns()
2. Call init() -- observe it reports memory-fallback
3. Call loadResults() -- observe it returns []

## Notes

Reproduced per review finding. Sources: Review 1 #1, Review 4 #2.
Lines: benchmarkStorage.client.ts:155, :190, :207, :251, :280

## Fix Plan

- Capture loaded rows before the retention-delete step
- On deleteRuns() failure, fall back to those captured rows rather than empty state
- Document in UI or logs that memory-fallback does not guarantee durability

## Resolution

Preserved retained benchmark history in memory before retention cleanup deletes; added regression coverage for init and load fallback paths.
