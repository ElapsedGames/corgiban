E) Acceptance Criteria (Measurable)
1. CompiledLevel matches ADR-0009: cell compaction, neighbor table, wall/goal bitsets, dead-square set, and goal-distance tables are built deterministically and unit-tested.
2. SolverState matches ADR-0009: boxes stored as sorted Uint16Array, occupancy bitset tracked, canonical player id is the minimum reachable cellId, and a 64-bit Zobrist hash + fingerprint are maintained. Zobrist tables are generated deterministically via seeded PRNG (no Math.random), visited/transposition tables are keyed by Zobrist with fingerprint verification, and no string keys are used in solver hot paths.
3. BFS solver is push-based, uses CompiledLevel/SolverState, expands reachability and pushes in canonical U, D, L, R order, honors cancel tokens and time budgets, and reports throttled progress; tests cover solve, cancel, and progress behavior.
4. analyzeLevel and chooseAlgorithm live in packages/solver/api/selection.ts, are exported, and have unit tests covering every rule branch and edge case (0 boxes, minimal grid, max boxes).
5. Solver entrypoint returns SolveResult with metrics including elapsedMs, expanded, generated, maxDepth, maxFrontier, pushCount, moveCount, and expands push sequences into a deterministic UDLR move string for playback.
6. normalizeSolverOptions enforces timeBudgetMs > 0, nodeBudget > 0, heuristicId/heuristicWeight restrictions for bfsPush, and heuristicWeight in [1, 10]; tests cover these bounds.
7. Worker runtime validates inbound/outbound protocolVersion/runId, SOLVE_START always carries a concrete algorithmId (never "auto"), SOLVE_PROGRESS includes bestPathSoFar only when enableSpectatorStream is true, and invalid messages are rejected with tests.
8. Worker client handles onerror/onmessageerror by setting workerHealth to 'crashed'; retry resets workerHealth to 'idle' and recreates the worker.
9. apps/web solverSlice stores recommendation/progress/status/workerHealth/result; SolverPanel shows recommendation, run/cancel controls, progress display, apply/animate solution actions, and a Retry button when workerHealth === 'crashed'.

G) Open Questions / Risks -- RESOLVED after codebase analysis

Q: How do we enforce time budgets and progress throttling without Date/Date.now in packages/solver?
A: Use worker-side time sources (performance.now) and pass elapsedMs into solver hooks; solver uses iteration counters for throttle intervals and never calls Date/Date.now directly.

Q: Where should solve orchestration live in apps/web?
A: Create solverPort + solverThunks; startSolve thunk runs analyzeLevel/chooseAlgorithm, stores recommendation, and dispatches worker calls. PlayPage renders SolverPanel and uses ReplayController for animation.
