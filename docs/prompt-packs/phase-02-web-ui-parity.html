F) Open Questions / Risks -- RESOLVED after codebase analysis

Q: Should /bench route be a minimal placeholder in Phase 2 or a more complete stub?
A: Route shell only -- export default + ErrorBoundary, placeholder heading. No bench logic in Phase 2.

Q: Does the dark mode class live on html or body, and how is it toggled?
A: On the html element (per docs/project-plan.md section 6). settingsSlice.theme drives it via a
   client-only ThemeApplicator component (useEffect + useSelector) in root.tsx. Initial theme
   changed from 'light' to 'dark' to match existing dark visual design (background: #0b1120).

Q: What are the initial level selection rules, and how is move history reset when switching levels?
A: Load index 0 from builtinLevels (from @corgiban/levels, parseLevel from @corgiban/core).
   nextLevelLoaded action wraps with modulo over totalLevels. History and stats reset to zero
   on every level load or level switch.

G) Implementation Notes (from codebase analysis)

- @corgiban/levels must be added as a runtime dependency to apps/web/package.json (currently absent).
  builtinLevels exports the classic Sokoban catalog. PlayPage reads index 0 on mount.

- LevelRuntime (Uint8Array/Uint32Array) cannot go in Redux. gameSlice stores JSON-serializable
  equivalents: playerIndex (number), boxes (number[]), initialPlayerIndex, initialBoxes (number[]).
  PlayPage holds LevelRuntime in a useRef; components reconstruct GameState on demand.

- gameSlice state shape:
    { levelId, levelIndex, totalLevels, playerIndex, boxes: number[],
      initialPlayerIndex, initialBoxes: number[], history: SerializableHistoryEntry[], stats }
  Actions: loadLevel, moveMade, undoDone, restarted (no payload -- uses stored initial fields),
           nextLevelLoaded.

- postcss.config must use .cjs extension (not .mjs) because apps/web has "type": "module".

- applyMoves(state, directions, options?): { state, changed, stoppedAt? }
  stopOnNoChange option stops early when a move has no effect and sets stoppedAt index.

- selectCellAt(state, index): { wall, target, box, player }
  Reads staticGrid for wall/target; scans boxes array for box; compares playerIndex for player.

- RenderPlan: { gridWidth, gridHeight, commands: DrawCommand[] }
  DrawCommand: { kind: DrawKind; col: number; row: number }
  DrawKind: 'wall' | 'floor' | 'target' | 'player' | 'playerOnTarget' | 'box' | 'boxOnTarget'
  buildRenderPlan iterates cells in row-major order (deterministic). draw uses color map only.

- vitest.config in apps/web uses environment: 'node'. gameSlice and buildRenderPlan tests are
  pure unit tests; no DOM or jsdom required. No config change needed.

- settingsSlice.ts: change initial theme from 'light' to 'dark' (1 line change).
