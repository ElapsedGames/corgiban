E) Acceptance Criteria (Measurable)
1. `packages/benchmarks` exports a benchmark domain model and runner with DB_VERSION/schema constants; migration tests cover version-upgrade paths and pass.
2. Benchmark run orchestration stores comparable metadata (algorithm/heuristic/budgets/environment) for every persisted result.
3. Benchmark worker pool concurrency defaults to `max(1, min(4, (navigator.hardwareConcurrency || 4) - 1))` and is enforced by tests.
4. BENCH protocol messages are run-scoped and mirror SOLVE semantics (`BENCH_START`, `BENCH_PROGRESS`, `BENCH_RESULT` keyed by `runId`); suite orchestration remains in `benchmarkClient` and `benchThunks`.
5. BENCH_* protocol messages are versioned/validated in worker schemas; discriminator union tests include BENCH_* and unknown-type rejection coverage.
6. Two-tier cancellation semantics are implemented and tested: keep `SolverClient` hard-reset behavior for `/play` (ADR-0013 unchanged), while benchmark suite cancellation uses `WorkerPool.cancel` for queued runs and `WorkerPool.dispose` plus generation-gated callbacks for suite cancellation. ADR-0014 documents the benchmark model.
7. Protocol-validation profiling records per-call parse cost and message frequency; progress-message gating is only allowed when total SOLVE_PROGRESS validation cost exceeds 2% of median benchmark elapsedMs. Structural messages remain strictly validated.
8. `/bench` provides suite builder, run/cancel controls, persisted results view, and IndexedDB-backed reload durability.
9. `navigator.storage.persist()` is feature-detected independently; diagnostics store `persistOutcome` as `granted | denied | unsupported`, denied is not treated as an error, and console logging is limited to dev/debug mode.
10. `performance.mark/measure` around solve dispatch/worker response are observed through PerformanceObserver and surfaced in a debug perf panel.
11. Benchmark report/level-pack export/import feature-detects File System Access APIs (`showSaveFilePicker`, `showOpenFilePicker`) and falls back to anchor-download; `fileAccess.client.ts` supports API injection for tests.
12. Solver budget defaults are validated at three layers: reducer clamp to positive values, UI input minima, and thunk fallback to `DEFAULT_SOLVER_TIME_BUDGET_MS`/`DEFAULT_NODE_BUDGET` when defensive guards detect invalid settings.
13. All affected tests are updated and `pnpm typecheck`, `pnpm lint`, and `pnpm test:coverage` pass.

G) Architect Decisions / Residual Risks
1. BENCH protocol granularity is resolved to per-run messages (`runId`-scoped), mirroring SOLVE flows. Residual risk: avoid suite-level protocol creep.
2. Cancellation contract is resolved as two-tier: keep ADR-0013 hard-reset semantics for interactive `/play`; benchmark pool uses queued cancel + suite dispose, documented in ADR-0014.
3. Validation-overhead gate is resolved to a numeric threshold: only gate SOLVE_PROGRESS validation when measured cost exceeds 2% of median benchmark elapsedMs.
4. Browser API variance is resolved via independent feature detection for File System Access and Storage Persist; unsupported/denied states are normal paths, not error states.
5. Stale callback risk is resolved with `suiteGeneration` gating in benchmark client progress/result dispatch plus explicit queued/in-flight race-path tests.
6. Budget validity risk is resolved with reducer clamp, UI min constraints, and thunk runtime fallback to solver defaults.

H) Sequencing Notes
- Implement benchmark domain + persistence schema first so worker and UI layers consume stable contracts.
- Lock BENCH protocol shape and benchmark cancellation ADR before wiring bench thunks/UI.
- Add suiteGeneration callback guards before enabling suite cancel in UI.
- Apply browser API feature-detection wrappers before export/import and persistence diagnostics tests.
- Land budget positivity defenses before switching `/play` defaults to settings-backed values.
